**Назначение приложения: Бэкенд для трекера задач**

Это приложение представляет собой серверную часть (бэкенд) для системы управления задачами.
Оно не имеет собственного пользовательского интерфейса, но предоставляет API (программный интерфейс) для его создания.

**Основные сущности:**

*   **Пользователи (Users):** Люди, работающие в системе.
*   **Задачи (Tasks):** Работа, которую нужно выполнить.

---

**Архитектура и жизненный цикл запроса**

Приложение построено на классической многослойной архитектуре. Давайте рассмотрим её на примере создания новой задачи.

**Шаг 1: HTTP-запрос от клиента**

*   Клиент (например, веб-браузер или мобильное приложение) отправляет `POST` запрос на специальный адрес (эндпоинт) `/api/tasks`.
*   В теле запроса содержится JSON с данными о задаче, например: `{"name": "Купить молоко", "description": "..."}`.

**Шаг 2: Контроллер (Controller)**

*   `TaskController` (класс в коде) принимает этот запрос.
*   Его задача — извлечь JSON из тела запроса и передать его на следующий уровень — в сервис. Контроллер не содержит сложной бизнес-логики, он работает как "диспетчер".

**Шаг 3: Сервис (Service)**

*   `TaskService` — это "мозг" операции. Он получает данные от контроллера.
*   **Обогащает модель:** Добавляет системные данные, которые не пришли от клиента:
    *   `id`: Генерируется автоматически.
    *   `createdAt`: Устанавливается на текущее время.
    *   `status`: Получает начальный статус `TODO`.
    *   `authorId`: Определяется на основе текущего аутентифицированного пользователя.
*   **Выполняет бизнес-логику:** Может проверять данные на корректность (валидация).

**Шаг 4: Репозиторий (Repository)**

*   Сервис передает полностью сформированный объект `Task` в `TaskRepository`.
*   Репозиторий отвечает исключительно за взаимодействие с базой данных. Он преобразует объект в команду для MongoDB.

**Шаг 5: База данных (MongoDB)**

*   `TaskRepository` отправляет команду на сохранение в MongoDB.
*   База данных сохраняет документ в коллекцию `tasks` и присваивает ему уникальный `_id`.

**Результат**

*   После успешного сохранения информация о созданной задаче (включая её новый `id`) возвращается по цепочке обратно клиенту в виде ответа на его HTTP-запрос.
